# 住宅会社仕様比較ツール — 機能仕様書

**バージョン**: 1.0  
**作成日**: 2026-05-13  
**対象**: Claude Code による実装

---

## 1. プロジェクト概要

### 背景と課題

注文住宅の建築検討において、以下の課題が生じやすい。

- 複数社（ハウスメーカー・工務店）の仕様を並べて比較するための場所がない
- 打ち合わせのたびに仕様が変更され、初期に決定した内容が埋もれる
- 「いつ・どの打ち合わせで・何を決めたか」の経緯が記録されない
- 仕様書と打ち合わせメモが別管理になり、整合性が取れなくなる

### 解決方針

打ち合わせ記録（議事録）と仕様比較を**一体型で統合管理**するブラウザアプリを構築する。  
打ち合わせで決まった事項を、そのまま仕様比較表に反映できる一貫したフローを提供する。

### 分離型を採用しない理由

「仕様は打ち合わせから生まれる」という構造上、分離すると二重入力が発生し管理が崩れる。  
また、仕様の変更経緯を後から参照できないという問題が起きる。

## 2. 技術仕様

| 項目 | 内容 |
|------|------|
| フレームワーク | React（JSX 単一ファイル） |
| データ永続化 | Claude Artifact Storage API（`window.storage`） |
| スタイリング | Tailwind CSS コアユーティリティクラス |
| 外部依存 | lucide-react（アイコン） |
| 動作環境 | Claude.ai Artifact（ブラウザ内） |
| エクスポート | JSON ダウンロード／インポート対応 |

### ストレージ設計

```javascript
// キー設計（Artifact Storage API）
"companies"        // Company[] — 会社一覧
"spec_items"       // SpecItem[] — 仕様項目一覧
"meetings"         // Meeting[] — 打ち合わせ一覧
"decisions"        // Decision[] — 決定事項一覧
"change_logs"      // ChangeLog[] — 変更ログ一覧
```

全データを5つのキーで管理し、読み書きは各キー単位で行う。  
セッションをまたいでデータを保持するため、`window.storage` を使用する（localStorage 不可）。

## 3. データモデル

### Company（会社）

```typescript
interface Company {
  id: string;           // UUID
  name: string;         // 会社名（例: "〇〇ホーム"）
  type: "maker" | "builder" | "other";  // ハウスメーカー / 工務店 / その他
  contact: string;      // 担当者名
  phone?: string;       // 電話番号
  email?: string;       // メールアドレス
  status: "considering" | "candidate" | "rejected" | "contracted";
  note?: string;        // 自由メモ
  createdAt: string;    // ISO 8601
}
```

### SpecItem（仕様項目）

```typescript
interface SpecItem {
  id: string;
  category: string;     // カテゴリ（例: "断熱", "窓", "設備", "構造"）
  name: string;         // 項目名（例: "断熱材の種類"）
  values: SpecValue[];  // 各社の仕様値
  createdAt: string;
}

interface SpecValue {
  companyId: string;
  value: string;        // 仕様の内容
  meetingId?: string;   // この値を決めた打ち合わせへの参照（省略可）
  updatedAt: string;
}
```

### Meeting（打ち合わせ）

```typescript
interface Meeting {
  id: string;
  companyId: string;    // どの会社との打ち合わせか
  date: string;         // YYYY-MM-DD
  location?: string;    // 場所（例: "展示場", "オンライン"）
  attendees: string[];  // 参加者（例: ["自分", "妻", "A担当：田中様"]）
  agenda: string;       // 議題（自由記述）
  summary?: string;     // 打ち合わせ全体のまとめ
  createdAt: string;
}
```

### Decision（決定事項）

```typescript
interface Decision {
  id: string;
  meetingId: string;        // 所属する打ち合わせ
  content: string;          // 決定内容
  specItemId?: string;      // 紐づける仕様項目（省略可）
  specCompanyId?: string;   // 仕様項目に紐づける場合の会社ID
  specValue?: string;       // 仕様比較表に反映する値
  status: "confirmed" | "pending" | "cancelled";
  note?: string;
  createdAt: string;
}
```

### ChangeLog（変更ログ）

```typescript
interface ChangeLog {
  id: string;
  specItemId: string;
  companyId: string;
  previousValue: string;
  newValue: string;
  meetingId?: string;   // 変更を決めた打ち合わせへの参照
  reason?: string;      // 変更理由
  changedAt: string;
}
```

## 4. 画面構成

### タブ構成（上部ナビゲーション）

```
[ 🏠 ダッシュボード ] [ 🏢 会社管理 ] [ 📋 仕様比較 ] [ 📅 打ち合わせ ] [ 📜 変更ログ ]
```

---

### 4-1. ダッシュボード

**目的**: 全体の進捗を一目で把握する。

表示内容:
- 検討中会社数 / 候補会社数 / 打ち合わせ総数
- 直近5件の打ち合わせ（日付・会社名・議題）
- 直近5件の決定事項
- 未確定（`status: "pending"`）の決定事項一覧（アクションが必要なもの）

---

### 4-2. 会社管理

**目的**: 検討対象の会社を登録・管理する。

#### 一覧表示
- 会社カード形式で一覧表示
- ステータスバッジ（検討中 / 候補 / 落選 / 契約済）
- 打ち合わせ件数を表示

#### 登録・編集フォーム
- 必須: 会社名、種別、担当者名、ステータス
- 任意: 電話番号、メールアドレス、メモ

#### 会社詳細
- 会社情報の表示
- その会社との打ち合わせ一覧
- その会社に関連する仕様値の一覧

---

### 4-3. 仕様比較

**目的**: 各社の仕様を項目ごとに横並びで比較する。

#### 比較テーブル

| カテゴリ | 項目名 | A社 | B社 | C社 |
|---------|-------|-----|-----|-----|
| 断熱 | 断熱材の種類 | 高性能グラスウール | ウレタン吹付 | — |
| 断熱 | 断熱等性能等級 | 等級6 | 等級7 | — |

- 会社が増えると列が増える
- 値が未入力の場合は「—」表示
- 値のセルをクリックすると編集できる（インライン編集）
- 値をクリックすると「この値を決めた打ち合わせ」へリンクが表示される（`meetingId` がある場合）

#### カテゴリフィルター
- カテゴリ別にフィルタリング（全表示 / 断熱 / 窓 / 設備 / 構造 / その他）

#### 仕様項目の追加
- 「+ 項目を追加」ボタンで新規項目を追加
- カテゴリ名は既存カテゴリから選ぶか、新規入力

#### 仕様値の反映フロー（重要機能）

打ち合わせの決定事項から仕様を更新する手順:

1. 打ち合わせ画面で決定事項に `specItemId` + `specValue` を紐づけ
2. 「仕様に反映」ボタンをクリック
3. 確認ダイアログで変更前後の値を確認
4. 確定すると `SpecValue` が更新され、`ChangeLog` に記録される

---

### 4-4. 打ち合わせ記録

**目的**: 打ち合わせの記録と決定事項を管理する。

#### 一覧表示
- 会社フィルターで絞り込み可能
- 日付降順で表示（最新が上）
- 件名・日付・会社名・決定事項件数を表示

#### 打ち合わせ登録フォーム

```
日付: [日付ピッカー]
会社: [会社ドロップダウン]
場所: [テキスト入力]（任意）
参加者: [テキスト入力] （カンマ区切り）
議題: [テキストエリア]
まとめ: [テキストエリア]（任意）
```

#### 決定事項の入力

打ち合わせ登録後、または編集時に決定事項を追加できる。

```
決定内容: [テキストエリア] ← 必須
ステータス: [確定 / 保留 / キャンセル]
仕様への反映:
  └ 仕様項目: [ドロップダウン]（省略可）
  └ 会社: [ドロップダウン]（省略可）
  └ 仕様値: [テキスト入力]（省略可）
メモ: [テキスト入力]（任意）
```

「仕様への反映」が入力されている場合、「仕様に反映」ボタンが表示される。

#### 打ち合わせ詳細

- 登録した全情報の表示
- 決定事項の一覧（ステータスバッジ付き）
- 各決定事項から、紐づいた仕様項目へのリンク

---

### 4-5. 変更ログ

**目的**: 仕様がいつ・どの打ち合わせで・何から何に変わったかを確認する。

#### タイムライン表示

```

  変更前: グラスウール 16K
  変更後: 高性能グラスウール 24K
  打ち合わせ: 2026-04-20 第3回打ち合わせ
  変更理由: 断熱等級6を達成するため仕様変更を依頼
```

#### フィルター
- 会社フィルター
- 仕様カテゴリフィルター
- 期間フィルター

## 9. ツール選択ガイダンス（Claude Code vs Cowork）

### Claude Code を推奨する理由

本ツールはブラウザ上で動作する React アプリ（単一 JSX ファイル）の実装であり、  
コードの生成・編集・テストが主作業となるため、**Claude Code が最適**。

| 観点 | Claude Code | Cowork |
|------|-------------|--------|
| 主な用途 | コード生成・編集・テスト | 非コーディングのファイル・ドキュメント作業 |
| 本タスクとの適合 | ✅ React/JSX の実装に最適 | ❌ コーディングは想定外の用途 |
| ターミナル操作 | ✅ 必要（npm, node 等） | ❌ 不要（GUIベース） |
| ファイル編集精度 | ✅ コード差分単位で正確 | △ ドキュメント向け |

### Cowork が向いているケース（本プロジェクトでの活用例）

実装後に以下の作業で Cowork を活用できる。

- 打ち合わせメモの PDF や Word ファイルから情報を抽出し、ツールに貼り付けるデータを整形する
- 各社のカタログ・仕様書 PDF を読み込んで、仕様比較表の初期データを作成する
- ツールから JSON エクスポートしたデータを Excel に変換して報告書を作成する

const STORAGE_WARNING_THRESHOLD = 400_000; // 400KB（文字数概算）

// 保存前チェック
async function saveWithCapacityCheck(key: string, data: unknown) {
  const serialized = JSON.stringify(data);
  if (serialized.length > STORAGE_WARNING_THRESHOLD) {
    showToast("warning", "データ量が多くなっています。不要なデータの削除を検討してください。");
  }
  await window.storage.setItem(key, serialized);
}

const SCHEMA_VERSION = "1.0.0";

interface StorageRoot {
  schemaVersion: string;
  migratedAt: string;

// 起動時マイグレーションチェック
async function runMigrations() {
  const meta = await window.storage.getItem("meta");
  const current = meta?.schemaVersion ?? "0.0.0";

  if (current === SCHEMA_VERSION) return;

  // マイグレーション定義
  const migrations = {
    "0.0.0->1.0.0": migrateV0toV1,
    // "1.0.0->1.1.0": migrateV1toV11,  // Sprint間追加
  };

  // 順次適用
  await applyMigrations(current, SCHEMA_VERSION, migrations);
}

// Sprint間でスキーマ変更が生じた場合の対応
// 例: Sprint2でcategoryをcategoryIdに変更する場合
async function migrateV0toV1() {
  const specItems = await loadSpecItems();
  const migrated = specItems.map(item => ({
    ...item,
    categoryId: await resolveCategoryId(item.category),
    category: undefined,  // 旧フィールド除去
  }));
  await saveSpecItems(migrated);
}

スキーママイグレーション

const SCHEMA_VERSION = "1.0.0";

async function runMigrations() {
  const metaRaw = await window.storage.getItem(STORAGE_KEYS.META);
  const meta = metaRaw ? JSON.parse(metaRaw) : { schemaVersion: "0.0.0" };
  
  if (meta.schemaVersion === SCHEMA_VERSION) return;
  
  // マイグレーション処理をここに追加
  // Sprint間でスキーマ変更がある場合は必ずここに記述する
  
  await window.storage.setItem(STORAGE_KEYS.META, JSON.stringify({
    schemaVersion: SCHEMA_VERSION,
    migratedAt: new Date().toISOString(),
  }));
}

function resolveIdConflict(importedItem, existingIds) {
  if (existingIds.has(importedItem.id)) {
    const newId = crypto.randomUUID();
    // 参照しているIDも再マッピング
    return { ...importedItem, id: newId };
  }
  return importedItem;
}
